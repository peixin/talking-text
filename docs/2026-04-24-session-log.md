## 阶段一：品牌与哲学（纯概念讨论）

### 1.1 产品定位的诞生

从一个深刻的痛点观察开始：**很多孩子花了大量时间学英语，但真正开口说的机会少得可怜。**

市面上通用的 AI 闲聊产品（如豆包等）虽然能陪聊，但**没有可控的学习范围**——对话中会随意蹦出孩子从未学过的词汇和超纲的复杂句型，导致孩子听不懂。挫败感一旦累积，原本轻松的“日常聊天”就变成了痛苦的“考试”。

于是我们提出了核心创意：

> **把孩子的英语课本喂给 AI，让 AI 只在这个已知范围内陪聊。**

产品的终极目标不是急于教给孩子大量新词汇，而是**让学过的词句真正沉淀并转化为孩子开口能用的能力**。

### 1.2 品牌命名的推敲

**英文名**：纠结过 `Talking Text` vs `TalkText`。我们选择了前者——`Text` 是主体，`Talking` 是状态，表达“会说话的课本静态文字”，而不是两个生硬的功能拼接。

**中文名**：由最初想法 `语滴`（寓意日常积累点滴）逐步提炼出 **`字有天地`**——四个字完美装进了维特根斯坦“语言的边界就是世界的边界”的哲学。字里有天地，认识多少字，你的天地就有多大。

**Slogan**：从“感知世界”一路演变升级到 **`言出成界`**：
- **言出** = 李白的“绣口一吐”（代表开口行动）
- **成界** = 维特根斯坦的“语言即世界”（代表行动带来的成长结果）
- 四字装下两个不朽的灵魂。

### 1.3 哲学双柱

维特根斯坦的“边界”与余光中/李白的“中心”，共同构成了我们的产品哲学：

- **维特根斯坦讲边界**：“我的语言的边界，就是我的世界的边界。”（选自《逻辑哲学论》，1921）
- **余光中替李白讲中心**：“绣口一吐，就半个盛唐。”（选自余光中《寻李白》，1980）

我们将这两件事完美结合为一件事：**绝不让 child 去剧烈撞击边界，而是让他安全、自信地站在自己世界的中心开口。边界，自然会一寸一寸往外推。**

### 1.4 Krashen 的 i + 1 理论

为了让孩子在已知范围里有成长，我们引入了语言学家 Krashen 的 **i + 1** 理论：真正高效的语言输入是“当前水平 + 一点点”。

因此，产品采用渐进式聊天机制：
- **~90%** 词汇来自已学词和句型——给足安全感，让孩子敢于开口。
- **~10%** 动态混入下一单元或同层级的主题新词——踮一下脚就能够到，在充足的语境中自然吸收。

这成为了 Scope Computer 内部 `stretch_vocab` 的设计源泉。

### 1.5 开屏词库

开屏标语中英文各自成立，而非逐字死译：

| 中文 | English |
|---|---|
| 开口，便是一个世界。 | Words become your world. |
| 开口一次，世界就大一寸。 | One word, one world. (含 `word` -> `world` 多一个字母 `l` 的视觉彩蛋，多出的一寸便是 `l`) |
| 用你懂的，说出你的世界。 | Your voice creates your world. |
| 说得出，才算你的。 | If you can say it, it's yours. |

## 阶段二：技术架构决策

### 2.1 产品形态：Web PWA

- **决定**：使用 Next.js Web App 承载，辅以 PWA，坚决不做 Native App。
- **理由**：App Store 上架与应用市场审核流程繁琐，多端适配（iOS/Android/鸿蒙）对于初创团队开发和维护成本过高。

### 2.2 技术栈核心选型

| 问题 | 核心决策 | 选型理由 |
|---|---|---|
| 语音与 LLM 供应商 | **火山方舟全家桶** | 火山引擎儿童 STT/TTS 音色优美，全栈统一有利于账号和结算闭环 |
| 主数据库 | **PostgreSQL 16** | 强一致性，且对 JSONB 支持极佳，未来易于扩展 pgvector 向量检索 |
| 前端框架 | **Next.js 16** | App Router 提供天然的落地页 SSR 极佳首屏体验和 SEO 优势 |
| 后端语言与框架 | **Python 3.12 + FastAPI** | 完美的异步协程支持，易于对接各类大模型与音频处理库 |
| 样式与 UI 组件 | **Tailwind CSS v4 + shadcn/ui** | 精简现代，拥有完全的源码控制力 |

### 2.3 关键架构原则

1. **适配器隔离 (Adapter Pattern)**：stt/llm/tts 是极易变动替换的供应商组件。所有外部第三方 SDK 调用必须放置于 `adapters/` 下，核心业务只依赖统一的 Protocol 抽象契约。
2. **Scope Computer 范围控制中枢**：每轮对话开始前，AI 必须向其获取“这一轮允许使用的词汇范围”。接口在 V1 阶段封死，内部实现将随产品演进（从 V1 的全大纲词汇 -> V2 的 i+1 -> V3 的 Mastery 掌握度跟踪）逐步升级。
3. **事件数据静默收集**：系统必须在每轮 Turn 对话时在后台写 `vocab_event` 记录（谁说了什么词，掌握度如何）。V1 阶段虽不读该表，但它是 V2 和 V3 掌握度分析算法赖以生存的基石。
4. **前端 SSR + CSR 混写原则**：首屏历史数据和应用外壳在服务端渲染，聊天界面的实时录音交互在客户端 Hydrate。
5. **代码遵循 Docker-ready**：虽然 V1 暂不编写多容器 compose 部署，但代码一律通过环境变量传参，杜绝在本地写死配置。

## 阶段三：项目脚手架搭建

### 3.1 仓库目录初始化
项目采用单仓库 (Monorepo) 组织，将前端 `frontend` 与后端 `backend` 并排放置于根目录下，并通过 `justfile` 统一驱动运行指令。

### 3.2 踩坑与踩实
1. **Poetry 2.0 包管理模式**：Poetry 2.0 默认项目需要可打包分发，我们在其 `pyproject.toml` 中配置了 `package-mode = false` 以适配纯 Web 应用后端。
2. **ESLint 9 Flat Config 稳定性**：由于 Next.js 16 最新的 Flat Config 兼容性在 ESLint 10 下会引发循环引用 bug，我们成功将其锁定在 ESLint 9 稳定版本，确保静态检查稳定。
3. **Tailwind v4 与朱红主题**：舍弃了默认的纯蓝/纯黑风格，在 `globals.css` 中将 `--primary` 前景色定制为大气的字有天地朱红色 `oklch(0.52 0.175 25)`，带给用户惊艳的第一视觉冲击。
