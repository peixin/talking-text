# 2026-05-27 开发日志 — 教材层级工作台与详细词表编辑器重构解耦

> 本次主要重构了 `/parent/materials/[groupId]` 教材详情编辑页的布局与交互。
> 目的是实现**层级管理与详细词汇表的分离**，拓宽输入框，增加子级目录的直观管理，并利用组件库规范联想输入。
>
> 不改数据库 schema，不改后端核心 API 行为，完美利用现有的 Server Actions 实现极速无刷新的高级感交互。

---

## 0. 起点 — 用户视角的问题与痛点

在之前的版本中，管理教材层级和词汇表时存在以下体验瓶颈：
1. **输入框极为局促**：采用三栏布局（左侧 1 栏层级基本信息，右侧 2 栏单词列表）。这导致极重要的层级路径输入框被挤压在左侧狭窄的区域，遇到较长的英文书名/单元名根本看不全。
2. **页面负荷过载**：层级深度关联、关联课本提示、AI 教学备注与单词表、短语表、句型表全部堆在一个页面上。对于真实的复杂教材，数据量过大导致版面冗长且容易造成操作混淆。
3. **层级输入不够顺滑**：使用 HTML 原生的 `<datalist>` 做模糊匹配联想，在移动端及复杂层级下体验十分生硬，没有明确的状态标志。
4. **子级节点管理缺失**：当家长点进 Level 1（例如 Kids Corner 教材）时，无法直观地看到并管理属于该教材的子章节（如 Book 1, Book 2...）。

---

## 1. 核心设计：双栏层级工作台 + 独立二级词表页

为了从根本上解决问题并提升整个系统的质感，我们采取了以下解耦与升级方案：

### A. 详细词句移入独立二级页面 ✅
* 新增二级路由：`/parent/materials/[groupId]/items` 专门用于词句管理。
* 将单词、短语、句型分别放置在 100% 宽度的三张精致卡片中。
* 独立编辑器采用宽屏视界，输入框长宽合理，极大减少了排版挤压。
* 顶部集成浮动固定的工具栏，提供 `← 返回教材信息` 及 `保存所有修改` 按钮。

### B. 重构主页面为 2:1 宽屏层级工作台 ✅
* `/parent/materials/[groupId]` 主页抛弃原有单词列表，整体改造成 `2/3` 与 `1/3` 的双栏布局。
* **左侧核心栏 (2/3)**：
  1. **当前节点标题直改**：在卡片顶部提供拥有下划线微动效的超大输入框，可一键直观修改当前章节的名字。
  2. **教材归属层级关联 (Levels Path)**：各级输入框拓宽到 100% 宽度，并集成组件库 Popover。
  3. **直属子章节目录 (Sub-chapters)**：动态渲染当前节点下的所有 active 子章节，提供直观的 Icon 与重点词数，并支持 **inline 失焦自动重命名**、**快速跳转**、以及 **"+ 快速添加子章节"** 的表单。
* **右侧辅助栏 (1/3)**：
  1. **重点词句掌握概况卡片**：使用渐变色和精致的三小圆环统计单词、短语、句型的当前总数，展示前 10 个词句作为预览大纲，并提供一键穿梭进入二级页的按钮。
  2. **教学元数据卡片**：承载关联课本来源及 AI 专属提示词。
  3. **彻底删除**：移入 Danger Zone。

## 2. 技术细节：组件库 Popover 与 inline 联动

本轮重构秉持 **"尽量使用组件库组件，减少手写代码量"** 的原则，做到了高复用与高兼容：

1. **集成组件库 Popover**：
   - 使用 `pnpm shadcn add popover` 命令，从项目中引入了基于 `@base-ui/react` 原语打包的 unstyled `components/ui/popover.tsx`。
   - 层级输入框将 input 作为 `PopoverTrigger`，聚焦或输入时在正下方弹起浮动匹配选项卡。
   - popover 的浮动位置计算、软键盘防遮挡、失焦自动关闭完全由底层组件库代劳，代码极为精简且支持无障碍导航。
   - 提供 “✓ 已存已有目录” 与 “+ 新建目录” 两种清晰选项，操作手感丝滑。
2. **子节点行内失焦自动保存**：
   - 子节点列表中的名字使用 `<input>` 渲染，监听 `onBlur` 事件及 `Enter` 键。
   - 当失焦或回车时，对比初始值，若有修改则直接无刷新触发 Server Action 的 `renameGroup(child.id, newName)`。
   - 命名变更实时合并进本地 React 状态，并调用 Next.js `router.refresh()` 缓存重刷，免去了繁琐的模态框，操作效率提升数倍。

## 4. 兼容性与 API 接口对齐

* **完全复用既有 Actions**：无需开发新的接口，全部复用已有 Server Actions：
  - 更新教材名称、提示词与层级路径：调用 `updateGroup(id, { name, source_book_hint, prompt_notes, levels })`
  - 更新单词/句型：调用 `updateGroup(id, { items })`
  - 子章节行内重命名：调用 `renameGroup(child.id, newName)`
  - 子章节快速添加：调用 `createGroup({ name, kind, parent_id })`
* **不影响其他模块**：Chat 练习流程、抽屉录入模块以及 LearnerProgress 的行为完好无损。

## 6. 后续优化方向 (Todo / Optimization)

在本次 Code Review 中，梳理出了以下几个在未来数据规模扩大或架构演进时可做的优化点：

1. **Next.js 路由缓存刷新优化 (Revalidation)**
   * **现状**：目前在更新/删除教材节点时，Server Action 中调用了对主详情页 `/parent/materials/[groupId]` 的 `revalidatePath`，但未同步刷新二级词表页 `/parent/materials/[groupId]/items` 以及子章节节点的 `/parent/materials/[childId]` 详情页。
   * **优化**：在 `updateGroup` / `renameGroup` 中补充对应的 `revalidatePath` 刷新，确保在强缓存环境下客户端路由数据绝对实时。

2. **前端 BFS 拓扑算法优化**
   * **现状**：`MaterialPickersClient.tsx` 中的 `descendantIds` 在前端采用了一个简单的循环 BFS 来获取后代节点，复杂度为 $O(N^2)$。当账号下教材节点只有几百个时性能极佳。
   * **优化**：如果教材库膨胀到数千个节点，可引入 React 渲染周期的邻接表索引缓存，将后代收集优化到 $O(N)$。

3. **后端 LanguageItem 联合匹配优化**
   * **现状**：后端 `_upsert_language_items` 函数中使用 `in_` 查询拉取 type 和 text 的交集，虽然在内存中进行了精确匹配过滤，但在有大量重名记录时 SQL 吞吐量会有多余的开销。
   * **优化**：未来可以考虑将其优化为使用 SQLAlchemy 的 `tuple_` 精确联合 `IN` 匹配，例如 `tuple_(LanguageItem.type, LanguageItem.text).in_(keys)`。
