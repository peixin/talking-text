# 2026-05-30 开发日志 — 拆分「采集」与「成书」，教材这块解套

> 本次解决长期困扰的"教材这块很混乱"问题。根因诊断、概念模型与三层设计写在
> [`content-lifecycle.md`](content-lifecycle.md)（中文版 [`content-lifecycle.cn.md`](content-lifecycle.cn.md)）。
> 本日志记录据此对代码做的重构（代号 B），以及随手清理的存量 mypy 债务。
>
> 不改数据库 schema。后端 API 行为有一处契约变更（`levels` → `tag_path`）。

---

## 0. 起点 — 为什么"混乱"

教材系统把两件本该分开的事压成了一件：**采集碎片**（每天拍照/输入）与
**拼装成书**（Book→Unit→Lesson 树）。旧实现让每张照片在采集那一刻就"自动归位"
进树——AI 推断 `parent_id`、把整库塞进 prompt、按位置猜 `kind`、`levels` 数组模糊
并树。归档这件事本质模糊、易错、又跟核心循环无关，却被做成了自动主路径，于是三个
症状同时出现：层级太复杂、UI 乱、概念模型没想清。

确立的原则（详见 content-lifecycle.md §1）：

> **采集是 append-only 且"笨"的；结构化是另一个、刻意的动作，由极少数人（先是运营者）
> 完成，绝不强加到每张照片上。**

并区分了两类用户（跟随者 / 贡献者）与"渐进式"的两个含义（消费渐进 / 创作渐进）。

## 5. 自查发现的存量缺口（非本次回归，待后续）

对 `create_group` / `update_group` / `_assemble_tag_path` 做了一遍正确性自查（未做运行时验证）。
未发现本次引入的 bug；以下两处是**沿袭旧 `levels` 分支的既有缺口**，建议后续补：

- **`tag_path` 路径无环检测**：非 `tag_path` 的 `parent_id` 分支有 `_check_cycle`，但 `tag_path`
  组装路径没有。实践中因 `_assemble_tag_path` 逐层从 root 精确匹配、很难构造出环，但理论上仍应补一道防护。
- **`tag_path` 路径不过 `_verify_write_permission`**：在 `locked` 的已有树下用 tag_path 追加内容时，
  不会触发锁校验（旧 levels 分支同样如此）。"在锁定树下新增内容"是否该受锁限制，需产品决定。

二者均与本次「采集/成书」拆分无关，且为既有行为，故本次未改，仅记录。

## 6. 整理工作台 V1 落地（content-lifecycle.md §4）

承接拆分后的「第 2 层」，实现了整理工作台的可用版本。设计岔路由用户拍板：
**收件箱 = 采集袋**、**归位 = 移动**、**首切片含练习派生候选**。

**后端**（`040fcb0`，加在 `groups` 路由，无 schema 改动）：
- `GET /organize/inbox` —— 当前 active learner 的散件，两个来源：① kind=`quick_practice` 采集袋的词；② **练习派生候选**：从该 learner 的 `turn.text_user` 分词，去掉停用词、去掉「已存在于该账号任何内容里的词」，按出现次数排序（上限 200）。守 Rule #3：候选是**实时派生**，不落表。
- `POST /organize/file` —— 原子**移动**：把词加入目标 tag 节点（幂等）+ 从来源采集袋移除；练习候选则惰性创建 `LanguageItem` 再归入。
- `POST /organize/dismiss` —— 把某采集词移出袋子（`LanguageItem` 行保留，可能被别处引用）。
- 复用 `_get_accessible_group` / `_verify_write_permission`（锁与订阅校验）/ `_upsert_language_items`。

**前端**（`6218e93`，`parent/organize/`）：
- 两栏页：左收件箱（采集袋词 + 练习派生新词），右 canonical tag 树（排除采集袋）。
- 交互：**点选散件 → 点右侧节点归位**（V1 用点选而非拖拽，零依赖、稳）；采集词可 ×（dismiss）；底部「新建标签路径」（`›`/`/` 分隔）走 `tag_path` 建节点，可同时把选中项归位，建完 `reloadWorkbench` 同步树。
- 入口：`parent/materials` 头部加「整理素材 →」链接。

**验证**：`just check` 全绿（含 tsc/eslint/prettier）。**未做运行时验证**（按规矩由用户发起）。

遗留/后续：拖拽交互（当前点选）；练习候选的「忽略」需持久化否则每次重现（V1 不做，候选不消失只是不烦人）；AI 提议成组（§4.2）；练习候选目前只取 `text_user`（孩子说的），AI 引入的 stretch 词暂未纳入。前端 organize 页用内联中文（与 parent/materials 邻居一致），i18n 化是后续债务。

## 7. 整理工作台改版（袋级 + AI 预判）+ 家长台/教材页产品化

用户反馈：一个词一个词点选太累；应**以「一次录入（采集袋）」为整体**来整理，且让 **AI 给预判/default、做到 ~80%、人只微调**（用户很懒）。据此改：

**后端**（`544a618`）
- `GET /organize/inbox` 改为**袋级**：返回 `capture_bags`（每袋含其词），不再是扁平 `capture_items`；练习候选不变。
- `POST /organize/suggest-bag` —— LLM 根据「袋内词 + 现有 tag 路径」预判一个 `tag_path`（优先复用已有路径）；任何失败**兜底为 [袋名]**，所以不依赖 LLM 可用性。
- `POST /organize/file-bag` —— 把整袋的词原子移动到 `tag_path` 叶子（复用 `_assemble_tag_path`），并归档空袋。

**前端**（`544a618`）：organize 页改为**一袋一张卡**：词预览 + 一个**预填 AI 预判路径**的输入框 + 「整袋归位」一键。挂载时并发为每袋拉取建议（兜底袋名），人改路径文字即可。练习候选改为**一个路径输入 + 全部归位**（用 `createGroup(tag_path, items=候选)` 一次入库）。

**教材页**（`aa8a299`）：`/parent/materials` 把 **canonical tag 树**（教材结构）与**未整理采集袋**拆成两块分开展示（之前混在一棵树里，导致「看不到 tag 树」）；采集袋块带「去整理 →」入口；`useKindLabel` 给 `tag`/`generic` 补上「标签」标签（新增 `Materials.kind_tag`，三语）。

**家长台**（`d88d16e`）：`/parent` 从简陋占位重做成产品化 hub —— 当前学习者卡 + 「录入→整理→对话」三步主流程卡（带实时计数：N 本教材 / N 袋待整理高亮）+ 孩子卡片 + 次级入口。

验证：`just check` 全绿。**均未运行时验证**（由用户发起）。`suggest-bag` 的 LLM 调用为外部依赖，已做兜底，但实际预判质量需真人在真数据上观察。
