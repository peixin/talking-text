## 阶段一：核心选型抉择

### 1.1 前后端协议选择：V1 HTTP，V2 升级 WebSocket

- **决策**：前端与 Next.js 代理之间走普通 HTTP 请求，后端 FastAPI 与火山引擎服务商内部消化网络等待。
- **理由**：
  - 在 V1 阶段，一轮对话的端到端耗时在 3-5 秒之间（语音识别 1s + LLM 推理 1-2s + 语音合成 1s），对于孩子来说是完全可以接受的起步体验。
  - 虽然基于全双工 WebSocket 的流式通信能将首个音频包返回降至 ~600ms 以内，但它的连接保活、中断重连以及前端音频播放状态机调试极其繁琐。
  - 核心架构铁律已规定：所有 Adapter 从第一天起就必须同时声明 `invoke()` 与 `stream()` 方法，因此 V1 我们以 HTTP Batch 形式开发，为 V2 的流式无缝升级预留了极佳的空间。

### 1.2 LLM 对接：拒绝绑定豆包 SDK，采用统一 OpenAI 标准协议

- **决定**：后端大模型适配器基于官方 `openai` 异步客户端开发，通过重写 `base_url` 指向火山方舟端点。
- **理由**：火山方舟的模型 API 100% 兼容 OpenAI 协议。这样在未来我们随时可以一行配置切换到 DeepSeek、Qwen、智谱等服务商，几乎不需要改动任何业务编排代码。
- **STT/TTS 特殊性**：针对没有行业标准的 STT/TTS，我们自定义 Protocol 契约，用纯 `httpx` 或 `websockets` 直连火山底层接口，彻底消除对臃肿第三方特定 SDK 的强依赖。

### 1.3 语音合成 (TTS)：选定火山儿童音色 Tina 老师 2.0

- **决策**：在火山合成音色列表中，挑选了专为中国儿童英语教学打造的 `zh_female_yingyujiaoxue_uranus_bigtts` 音色。它不仅完美支持英式英语发音与中英双语混合播报，而且资费更加便宜，非常契合口语陪伴场景。
- **API 选项**：选用 HTTP Chunked（分块流式传输）端点，使后端能以单向数据流形式持续累积 base64 音频并一次性回传给前端播放。

### 1.4 音频编码重封装：浏览器 WebM/Opus → 后端重封装为 Ogg/Opus

- **原理**：各现代浏览器（除 Safari 外）均原生支持 `webm/opus` 录制，其 10 秒录音体积仅为 25KB 左右，非常利于移动网络上传。但火山 STT 底层只接收规范的 `ogg/opus`。
- **方案**：后端拦截上传的 `webm` 文件，利用 `ffmpeg -c:a copy -f ogg` 命令在 **~30毫秒** 内仅做容器重封装而不进行高耗时的重新编码，实现零质量损失、超低延迟的格式转换。

## 阶段二：数据建模的核心博弈

### 2.1 Turn 对话记录为什么必须采用物理关系表，而非 JSON 字段？

在建模讨论中，团队曾提出Turn数据更偏向日志，是否能直接作为一个大 JSON 数组挂载在 Session 下。我们予以坚决回绝，坚持**按轮单行建表**，理由如下：

1. **掌握度追踪必须外键引用 Turn ID**：Mastery 算法在分析词汇表现时，必须追踪“该词是在第几轮、哪个上下文中被说出的”。如果 Turn 只是 JSON 数组中的非固定元素，我们将无法在 `vocab_event` 中建立稳定的外键引用。
2. **高效的财务与计费统计**：家长在后台查询 Token 消费和时长账单时，使用标准的 SQL 联查和 `SUM` 聚合在数万行数据中只需毫秒级，而解析大体积 JSONB 数组将对数据库造成毁灭性打击。
3. **部分执行失败的事务原子性**：单轮对话如果 STT 成功但 LLM 异常，本轮 Turn 不会落盘，这保证了关系表的绝对干净。

### 2.2 核心资产：`vocab_event`（词汇事件静默收集）

我们坚持从 V1 第一天起就静默捕获每次 Turn 产生的单词事件（AI 说了什么词，孩子用了什么词，说话者是谁）。  
**这绝不是装饰品，而是整个产品的数据基建投资**。假设孩子连续使用了 3 个月，每天交互 50 轮：
- **如果不收集**：V2 mastery 掌握度系统上线时，面对这个孩子，其掌握度词表为零。我们必须重新收集 3 个月才能得出结论。
- **如果收集了**：V2 上线的一瞬间，一条 SQL 便能把孩子过去 3 个月所有听过、说过、错过的词汇数据梳理得清清楚楚，掌握度模型立即可用。

## 阶段三：开发落地清单

1. **后端数据库迁移**：新建 `turn` 表与 `vocab_event` 表，继承自 `TimestampMixin`。
2. **适配器测试烟囱化**：STT、LLM、TTS 适配器均编写了 `if __name__ == "__main__"` 自闭环单体运行脚本，支持开发期各组件离线调试。
3. **前端麦克风状态机**：`ChatClient.tsx` 中编写了严密的麦克风点击录制逻辑，包含 `idle` (空闲)、`recording` (录音中)、`uploading` (上传中) 三态，且返回的 Base64 自动拼接 Data URL 并送入浏览器 Audio 标签秒开播放。
4. **前端 Multipart 边界踩坑修复**：修复了 `lib/backend.ts` fetch 封装时默认强行添加 `application/json` 请求头的 Bug。对于 `FormData` 传输，必须将 `Content-Type` 设为空，以便浏览器能够自动插入独特的 multipart boundary。
