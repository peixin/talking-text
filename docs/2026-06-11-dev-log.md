# Dev Log (2026-06-11) — Child safety & input limits · Material sharing

Two work blocks today. First, pre-Phase-1 hardening: before inviting
non-founder families (roadmap Phase 1), the product needed two guardrails —
the AI must never follow a child into an unsafe topic, and every input path
needed a size ceiling. Second (§4, after the deploy went live and families
were invited): private material sharing between parents, so invited families
with the same school progress can use the founder's prepared book.

---

## 1. Content safety — always-on prompt rules

**Approach (three layers, one implemented now):**

1. **System-prompt safety section** (this change) — the primary defense.
2. **Vendor mandatory moderation** — DeepSeek / Doubao / Qwen are domestic,
   regulated models with built-in content filtering (especially political
   topics); raw model output is already filtered once. Free second layer.
3. **Dedicated moderation API** (Volcengine / Aliyun content safety) on input
   and output — adds cost + latency; deferred to public launch alongside PIPL
   work.

**Implementation:** `_SAFETY_INSTRUCTIONS` in `core/prompt/assembler.py`,
injected as the second section (right after the persona) in **every** mode
(calibration / free / group) and under custom personas. Rules:

- Never discuss / describe / role-play: sexual or romantic content, violence,
  politics, religion, drugs/alcohol/gambling, horror, death, private body parts.
- **Child brings it up** → don't engage, explain, or scold; one light sentence,
  redirect to a safe topic.
- Child seems sad/scared/hurt → one kind sentence + suggest talking to parents
  or teacher.
- Never ask for or repeat personal info (address, school, phone, IDs); never
  suggest meetups, purchases, or external sites/apps.

Two tests lock the section into all modes and its position after the persona.

## 2. Input limits — `config.toml [limits]`

New `[limits]` section + `LimitsConfig`. **Backend is authoritative** (every
request checked); the frontend mirrors the values in `lib/constants.ts` as UX
guards (same pattern as `SCOPE_SOFT_CAP`) — keep the two in sync when tuning.

| Key | Default | Backend | Frontend |
|---|---|---|---|
| `chat_text_max_chars` | 500 | 422 `TEXT_TOO_LONG` | textarea `maxLength` + counter |
| `chat_recording_max_seconds` | 60 | — (byte cap backstops) | auto-stop + countdown |
| `ingest_text_max_chars` | 10000 | 400 | `maxLength` + counter |
| `ingest_max_images` | 5 (was hardcoded) | 400 | counter badge + disabled buttons |
| `ingest_image_max_mb` | 10 (was hardcoded) | 400 | pre-check + warning toast |
| `ingest_recording_max_seconds` | 120 | — | auto-stop + countdown |
| `audio_upload_max_mb` | 10 | 422/400 on all audio uploads | — |

Sizing notes:

- Recording 60 s (child turn) / 120 s (parent may read material aloud). At
  webm/opus bitrates 120 s ≈ 0.5 MB, so the 10 MB audio cap is purely an
  anti-abuse backstop.
- `ingest_text_max_chars = 10000`, not 5000: the **re-extract flow round-trips
  the full-page OCR transcription** through the description field; a dense
  textbook page can exceed 5000 chars.
- AI reply length needed no work — already capped by `[task.dialog]
  max_tokens = 200` plus the persona's ≤ 15-word rule.

Refactor on the way through: batch and streaming turn endpoints had identical
audio read/transcode blocks; extracted into a shared `_read_turn_input()` in
`api/session.py`, so the limit checks live in one place.

## 3. Client-side feedback (no silent walls)

- **Char counters** on chat input + ingest description — appear only at ≥ 80%
  of the limit (`412/500`), invisible otherwise.
- **Recording countdown** — surfaces only in the last 10 s ("auto-stops in 8s"
  under the chat mic; `(8s)` on the ingest stop button). Deliberately not a
  full-session timer: a visible clock pressures the child, which contradicts
  "make them comfortable speaking".
- **Image cap** — `n/5` counter badge next to the upload buttons (amber when
  full), photo/upload buttons disabled at the cap; the warning toast remains as
  backstop for a single over-sized multi-select. Toast texts now take `{max}`
  params, so tuning the config doesn't touch translations.
- New i18n keys (`Chat.recording_countdown`, `Ingest.image_count`) in en /
  zh-CN / zh-TW.

**Known gap (accepted):** voice-transcribed text is appended to the ingest
description programmatically, bypassing `maxLength`; the backend cap catches
the extreme case with a visible error.

## 4. Material sharing — private book sharing between families

**Context:** deployed to production and invited classmates' parents (same
school progress). The founder's prepared book needs to reach them — but NOT
via a public library (copyright); only private, link-based sharing between
consenting parents.

**Design already existed** — `learner-content-scope.md` (2026-05-27) had
specified the whole feature, edge cases pre-decided (§6 UC-5/6/7, §7, §8.4),
and half the schema was already live (`item_group_subscription` with tombstone
semantics, `cloned_from_group_id`, `locked`). This change implemented the
remaining half. Co-building (multiple parents editing one book) stays deferred
per §2.7 — the subscription model is its forward-compatible precursor
("one maintainer + N followers"), nothing to undo later.

**Semantics (receiver chooses at adoption, link encodes no mode):**

- **Subscribe** (default, highlighted) — live reference, no copy; owner's
  edits propagate; read-only for the receiver.
- **Clone** — deep copy, independent ever after.
- **Fork** — a subscriber turns the subscription into an editable clone;
  learner assignments are re-pointed in the same transaction. Mastery stats
  survive because membership re-points to the SAME canonical `language_item`
  rows (keyed by (type, text)) — no stat migration needed, ever.
- Books only (root groups); subscribers cannot re-share (create requires
  ownership, §7.3). Revoking a code blocks new adoptions only; cutting
  existing subscribers off = deleting the book (tombstone, §7.1).

**Backend** (migration `c62cb152ead5`):

- `group_share_link` — 8-char code from an unambiguous alphabet (no 0/O/1/I/L),
  idempotent per book, revocable. `group_adoption` — clone provenance
  (surrogate PK + UNIQUE(target); the doc's composite PK was invalid since
  `source_group_id` is SET-NULL-able).
- `app/api/share.py` — create/revoke link, unauthenticated preview,
  adopt (subscribe|clone), fork, unsubscribe, list subscriptions (incl.
  tombstones). `core/sharing.py` — subtree deep copy.
- `GET /groups` now returns subscribed trees with `subscribed: true`, so the
  scope picker and session flow work on subscribed books unchanged.
- **Bug fixed in passing:** `_get_accessible_group` only honored subscriptions
  on the exact group id — child units of a subscribed book 404'd. It now walks
  the parent chain to the root before checking.

**Frontend:**

- Materials list: share button on owned roots (copies
  `/{locale}/parent/materials/share/{code}` + shows the code), "add a shared
  book" paste box (accepts URL or bare code), subscribed badge + hidden edit
  controls, fork/unsubscribe on subscribed roots, tombstone rows.
- Landing page `parent/materials/share/[code]` — preview card + the two
  adoption choices (static segment wins over `[groupId]`, no route conflict).
- Detail page: subscribed banner; writes are backend-403 backstopped (the
  1459-line `GroupDetailClient` was not deep-refactored for read-only — later).

**Known gaps (accepted):** login redirect does not preserve the share URL
(receiver can paste the code in the materials page instead); detail-page edit
buttons remain visible on subscribed books until the read-only refactor.
