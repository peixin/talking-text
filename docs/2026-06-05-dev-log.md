# 开发日志 / Dev Log (2026-06-05)

本日志记录适配器层的一次系统性整理：存储抽象（BlobStorage）、LLM 适配器按能力归一、按交互环节配模型，以及材料抽取的两段式原型。主题是同一条纪律——**`core` 只依赖 Protocol，供应商差异收敛成配置**。

涉及提交：`5170cca`（存储）· `ce33842`（LLM 协议）· `5440e3b`（按环节配模型 + 两段式抽取）。

---

## 1. 存储抽象 BlobStorage / Store keys, not paths（`5170cca`）

**问题**：录音音频以**绝对本地路径**写进 `turn.audio_in_path` / `audio_out_path`。一旦换机器、上容器、或迁云，全表路径作废；且 `core` 直接 `pathlib` 读写、读 `AUDIO_STORAGE_DIR`，违反适配器纪律（规则 #1）和 Docker-ready（规则 #8）。

**改动**：
- 新增厂商中立的 `BlobStorage` Protocol（`app/adapters/storage/protocol.py`）：`put / get / exists / delete / url`。云后端 `url(key)` 返回**签名 URL**（浏览器直连），本地后端返回 `None`（由我们自己的鉴权端点吐字节）。
- `LocalBlobStorage`（本地盘，带路径穿越防护）+ `NullBlobStorage`（关闭持久化时 no-op）。
- **DB 改存 storage key 而非路径**：`audio_in_path/out_path` 现在是后端无关的 key（`{learner}/{session}/{turn}_{dir}.{ext}`）。列名保留（避免 schema 变更）。本地开发的 33 条存量绝对路径已一次性转成 key（本地阶段，迁移脚本跑完即删，未进 alembic 历史）。
- `DialogOrchestrator` 注入 `blob`；`get_turn_audio` 抽出 `_serve_stored_audio`：有签名 URL 就 302，否则读字节返回。`core` 不再碰 `pathlib` / `AUDIO_STORAGE_DIR`。

> 取向（记给未来）：以后挑便宜的云对象存储（七牛/阿里/腾讯/火山 TOS / MinIO）各写一个 `BlobStorage` 实现即可，可迁移；对服务而言它就是「一个带鉴权的 URL 前缀」。

## 2. LLM 适配器按能力归一 / Text & Multimodal role protocols（`ce33842`）

**问题**：`LLMAdapter` 是个**胖接口**，强制每个实现都有 `invoke_vision`，文本模型（DeepSeek）只能抛 `NotImplementedError`。而 `VolcLLMAdapter` 与 `DeepSeekLLMAdapter` ~90% 重复——都是 `AsyncOpenAI` 客户端 + 一样的 invoke/stream。

**改动**：
- **按输入能力拆角色 Protocol**（接口隔离）：
  - `TextLLM` — `invoke` / `stream`（文本进文本出）
  - `MultimodalLLM` — `TextLLM` + `modalities`（配置的模型还能吃图片/音频/文档）
  
  调用方依赖**最窄角色**：对话/标题/掌握度/校准依赖 `TextLLM`，抽取依赖 `MultimodalLLM`。`invoke_vision` 及其 `NotImplementedError` 桩消失，图片现在是 `LLMMessage.content` 里的 `ImagePart`（OpenAI content-parts 形态）。
- **一个实现收编所有厂商**：`OpenAICompatibleLLMAdapter(api_key, base_url, model, modalities, extra_body, reasoning_effort)`。火山 Ark / DeepSeek / 阿里百炼 / 腾讯混元 / 小米 MiMo 全是 OpenAI 兼容，差异只是配置。DeepSeek 的 `thinking`/`reasoning_effort` 降级成构造参数（`extra_body`），不再是独立类。删除 `volc.py` / `deepseek.py`。
- 能力在**启动时校验**（fail fast）取代运行时 503。
- STT / TTS 保持独立类别（协议不同，非 chat-completions）。

## 3. 按交互环节配模型 + 两段式抽取原型（`5440e3b`）

### 3.1 按环节配模型 / Per-stage model config

`config.toml` 改成 stage 表，每个 AI 交互环节独立选 provider + model——高频对话用便宜模型，材料整理用强模型：

```toml
[adapter.stage.chat]          # 对话 + 工具任务(标题/掌握度/校准)
provider = "deepseek"
model = "deepseek-v4-flash"
context_window = 1048576
thinking = "disabled"

[adapter.stage.extraction]    # 单次多模态抽取(见 3.2)
provider = "volc_ark"
model = ""                    # 火山 vision endpoint 来自 VOLC_ARK_VISION_MODEL(.env)
```

新增 **阿里百炼（DashScope）** 与 **小米 MiMo** 两个 OpenAI 兼容 provider（base_url + key 在 `.env`）。四家共用 `OpenAICompatibleLLMAdapter`。加新厂商 = `.env` 加 key + `factory._openai_llm()` 加一个 `case` + config 指过去。

**选型与定价（截至 2026-06，点位价）**：
- **对话** → DeepSeek `deepseek-v4-flash`：缓存命中 ~¥0.02/M 是关键——静态 system prompt（教材+人设）每轮重复 → 多轮几乎全缓存命中。英文简单问答简单模型即可。
- **材料整理** → 需多模态 + 推理，建议阿里 `qwen3-vl-plus`（拿到 key 后切）；暂默认火山 doubao vision（不破坏现状）。
- **STT** → 保持火山（处理孩子中英混说）。备选 Qwen3-ASR（OpenAI 兼容 `/audio/transcriptions`）。
- **TTS** → 保持火山 Tina（少儿英语教学音色）。小米 TTS 当前免费，可后续 A/B。

> 抽取低频，成本是噪声；按**质量 + 可维护性**选，而非省钱。

### 3.2 两段式抽取原型 / Two-stage extraction

材料整理是否该「先归一成文本，再由强文本模型结构化」？讨论结论：**裸 OCR 会丢版式语义**（少儿教材的图↔词配对、色块分组是关键信号），但只要 perception 是**版式感知转写**而非裸 OCR，「一个强文本大脑统一结构化」就成立，是好工程（可维护性大赢）。

落地为一个可 A/B 的开关（`config.toml [adapter.ingest] extraction_mode`）：

```toml
[adapter.ingest]
extraction_mode = "single"   # 或 "two_stage"
```

- `single`（默认，行为不变）：一次多模态调用做 OCR + 结构化。
- `two_stage`：
  1. **perception**（VLM）把页面转成版式感知 Markdown，强制保留图↔词配对 `[picture: a red apple] → "apple"`、表格、色块分组；
  2. **structuring**（`deepseek-v4-pro` thinking，唯一文本大脑）把转写文本 → language item。
  - 纯文本输入跳过 perception，直接进 structuring。

默认值让 two_stage 用现有火山+DeepSeek key 就能跑。**命门是转写保真**：`perf_logging` 已开，日志里能看到 perception 全文转写，需在真实课本页上眼检图↔词配对有没有丢——这只能拿真实材料验证。

> 原型暂放 `app/api/ingest.py`（与既有抽取同处）。未来做「材料重整理 / 未入库素材整理 / 用 DB 已有数据让 AI 自动归到某课本某章节」的闭环时，再把抽取编排抬进 `core/curriculum/`（带 DB 上下文），并把 structuring 这个「唯一大脑」扩展成归档建议器。perception 的转写产物天然是 capture 原始件，可存下来重跑 structuring 而不重新 OCR——契合 capture/canonical 生命周期。

## 4. STT / TTS 多厂商适配器

把语音链路也做成可选 provider。研究各家接口形态后，**只实现接口清晰、能放心写的**，不照想象盲写：

| 角色 | provider | 接口形态 | 状态 |
|---|---|---|---|
| STT | `volc` | WebSocket | 既有 |
| STT | `dashscope` | Qwen3-ASR 走 compatible-mode `/chat/completions` + 音频 content part（中英混说自动识别） | ✅ 新增 `DashScopeQwenASRAdapter` |
| TTS | `volc` | HTTP chunked（Tina 少儿音色） | 既有 |
| TTS | `openai_compatible` | 标准 `/audio/speech`（OpenAI / 自托管 Qwen3-TTS） | ✅ 新增 `OpenAITTSAdapter` |

切换：`config.toml [adapter] stt_provider` / `tts_provider`；密钥/端点在 `.env`（`DASHSCOPE_*` / `OPENAI_TTS_*`）。

**有意未实现（bespoke / 文档不全，不盲写）**：
- 阿里 **Qwen-TTS**——非 OpenAI，DashScope 原生 WebSocket realtime，需对协议实测。
- 小米 **ASR/TTS**——公开文档无具体接口形态（ASR 开源权重，TTS 文档 JS 渲染）。

**待验证的坑（线上未测）**：
- Qwen-ASR 文档示例是 wav/mp3，而管线现送 **ogg/opus**——若不被接受，需在送 STT 前转 wav/mp3。
- `dashscope` 下 `stt_audio_seconds` 记 0（Qwen-ASR 是 token 计费，非按秒）。
- TTS 的 voice 仍读 `VOLC_TTS_DEFAULT_VOICE`（通用 voice 的历史耦合）——切 `openai_compatible` 时要把它改成对应端点的 voice 名（如 `alloy`）。

## 5. 模型注册表 + 任务参数 + 去 hardcode（参考 diting）

把模型元数据（上下文/能力/价格）抽成一份注册表，支撑上下文判定、能力门控、以及未来的计价；同时把散落的 `max_tokens`/`context_window` hardcode 收编成配置。

### 5.1 `models.toml` —— 按渠道嵌套
```toml
[defaults]
context_limit = 131072                 # 未列模型的回落

[model.deepseek."deepseek-v4-flash"]   # 渠道是表层级，模型名干净无前缀
kind = "llm"
context_limit = 1_000_000
features = ["cache", "thinking"]
input_modalities = ["text"]
output_modalities = ["text"]
pricing = { unit = "token_million", input = 1.0, output = 2.0, cache_hit = 0.02 }
```
- **匹配键 = (provider, model)**，正是 `config.toml [adapter.stage.*]` 的两个字段；`app/model_registry.py` 按 `(provider, model)` 元组查。
- **渠道做成表层级**而非名字前缀——同名不同渠道（`deepseek` 直连 vs `aliyun` 中转，价格不同）天然分两节，零前缀冲突。
- `pricing.unit` 覆盖异构计费：`token_million`（LLM，可阶梯 `tiers`）/ `char`（TTS）/ `audio_second` / `audio_hour`（STT）/ `free`。比 diting（仅 LLM token）富。
- 注册表是**元数据叠加层**：未知模型 `context_limit` 回落 `[defaults]`——能先用 probe 试没登记的模型。
- 决策记录见命名讨论：选「config 留 provider+model，注册表按渠道嵌套」。

### 5.2 火山接入点（接入点 / endpoint）映射
火山的 `ep-xxxx` 是部署层间接（同 Azure OpenAI deployment / SageMaker endpoint），有流量优惠但污染模型身份。解法：**模型名是身份，接入点是可选「线上 id」覆盖**。
- `.env VOLC_ARK_ENDPOINTS`（JSON：逻辑名→`ep-xxxx`，账号相关）。有则用接入点，无则直接发模型名。
- factory volc_ark 分支：`wire_model = endpoints.get(model, model)`，对任何 volc_ark 模型生效。
- config/registry/计费 全程只见 `doubao-vision` 这类逻辑名；`ep-xxxx` 只活在 .env 一行。删掉了旧的单值 `VOLC_ARK_VISION_MODEL` 特例。

### 5.3 去 hardcode
- **`context_window`**（旧时 stage 里手敲 1048576/262144）→ 删除，`session.py` 硬限改查 `model_registry.context_limit(provider, model)`，换模型自动跟着变。
- **`max_tokens` / `temperature`**（8 处：orchestrator/session/calibration/mastery/learner/groups/ingest×3）→ 收编进 `config.toml [task.*]`（`dialog/greeting/title/calibration/mastery/persona/group_naming/extraction/perception`），读 `app_config.task(name)`。任务决策与模型属性分离。
