# 内容生命周期：采集 → 成书 → 公共库

> 设计文档 — 2026-05-30 定稿。
> 通过把当前代码**揉在一起的两件事**拆开来，解决"教材这块很混乱"的问题：
> **采集碎片** 与 **拼装成书**。
> 位于 [`content-model.md`](content-model.md)（存储 schema）与
> [`learner-content-scope.md`](learner-content-scope.md)（多 learner / 分享）之上。
> 英文为准版：[`content-lifecycle.md`](content-lifecycle.md)。

---

## 1. 本文确立的核心区分

当前实现试图让"每天一张照片"在**采集那一刻就自动归位进一棵完整的教材树**（AI
推断 `parent_id`、把整个库塞进抽取 prompt、靠位置猜 `kind`、`levels` 自动建树并
模糊并树）。这正是三个症状的**唯一总病根**：*层级太复杂、教材 UI 乱、概念模型没想清。*

修法只有一条原则：

> **采集是 append-only 且"笨"的。结构化是另一个、刻意的动作，由极少数人
> （早期就是运营者你自己）完成，绝不强加到每张照片上。**

"渐进式每日添加"**不等于**"每张照片自动归位进树"。它的意思是：碎片在采集流里
一天天**累积**；而*变成一本结构化的书*，是另一次整理动作的产物。

---

## 2. 两种用户（用法完全不同）

绝大多数用户从不采集、从不整理。产品必须**先服务跟随者，再服务贡献者**。

| | **跟随者（绝大多数）** | **贡献者（极少数 + 运营者）** |
|---|---|---|
| 目标 | 挑一本现成教材，让孩子跟着学 | 把碎片建成干净的书并发布 |
| 碰**采集**吗 | 从不 | 是 |
| 碰**结构/树**吗 | 从不 | 是 —— 他们是第 2 层唯一的用户 |
| "渐进式"对他意味着 | 一本现成的书，单元每天**解锁**（节奏） | 碎片一天天**攒**进在建的书里 |
| 编辑权 | 只读（订阅）；见 `learner-content-scope.md` §9.4 | 对自己的 canonical 书有完整权限 |

设计结论：`ingest.py` / `groups.py` 里那套精巧的自动归档逻辑服务的是**贡献者**
——一个极小的群体——却被做成了所有人的默认主路径。而绝大多数人真正需要的
（一个塞满了书、可供 follow 的库）在运营者去填充之前**根本不存在**。

## 3. 三层模型

```
┌─ 第 1 层 · 采集流 (Inbox)─────────────────────────────────────┐
│  per-learner，append-only，扁平，零归档决策                    │
│  拍照 / 粘贴 → AI 只抽【词】（不分类）→ 落进"<learner> 的采集"  │
│  孩子可以【立刻】对着采集词袋开口                              │
│  → 这就是验证核心循环所需的全部                                │
└───────────────────────────┬───────────────────────────────────┘
                            │  贡献者刻意"整理"：把词拖进 tag 树
                            ▼  （不是拍照时自动猜）
┌─ 第 2 层 · Canonical 教材 (tag 树)────────────────────────────┐
│  一棵单父的【tag】树（Tot Talk › Book 1 › Unit 1）            │
│  这才是那棵"完整的树"。它是【整理的产物】，不是采集的副作用     │
│  详见 §4。                                                    │
└───────────────────────────┬───────────────────────────────────┘
                            ▼  整理干净 → 发布
┌─ 第 3 层 · 公共素材库 (V2)────────────────────────────────────┐
│  `owner_account_id = NULL` 的 canonical 教材                  │
│  跟随者订阅；单元渐进式解锁                                    │
│  设计种子已在 learner-content-scope.md §2.7、§12               │
└───────────────────────────────────────────────────────────────┘
```

### 为什么现有 schema 已经完美支持——不用重构

`LanguageItem` 是全局去重的（`UNIQUE(type, text)`）。因此"把采集到的词提升进
canonical 书的 Unit 3"只是**插一行 `item_group_member`——零拷贝**。同一行
`apple` 同时被采集袋和 canonical 课次共享。"万物皆组"的模型本来就是为这个造的，
只是你还没这么*用*它。

## 4. 第 2 层细节 —— 整理工作台

这是整理动作的具体规格。它是**唯一**建层级的地方，由贡献者（早期是运营者）操作。

### 4.1 节点是 TAG，不是 level

用户脑子里没有"第 1 层 / 第 2 层 / level 3"。他认识的是名字：`Tot Talk`、
`Book 1`、`Unit 1`、`Lesson 1`。所以：

- **每个 `item_group` 节点就是一个 tag** —— 一个名字。没有类型化的结构角色去驱动树。
- **深度自由。** `Tot Talk › Book 1 › Unit 1 › Lesson 1`（4 层），或更短，或单独一个
  `General Practice`（1 层）。不强求"必须是 书/单元/课"。
- **单父树**，不是平行多标签。理由：产品的层级是真正的包含关系（一课只属于一个
  单元）、顺序有意义（Unit 1 在 Unit 2 前；上册在下册前）、且 Scope 的子孙遍历
  （`get_descendant_group_ids`）本来就假设单父。你给的例子本身就是一条干净单链。
- **`kind` 降级为装饰标签。** 它可以带上 AI 的 `kind_label`（"教材"/"单元"/"课次"）
  作为 tag 旁边的小灰字提示，但**绝不驱动结构、绝不被 Scope 读取**。按位置猜
  `kind`（`idx == total-1 → lesson`）—— "level N" 思维的化身 —— 删掉。
- **用户看到的**就是 parent 链上一串 tag 名字的面包屑。他永远看不到 level 编号。
- **主题横切标签**（`animals`、`童话`）明确推迟到 V2。V1 只做层级 tag 路径。

### 4.2 工作台：散件收件箱 → tag 树

```
整理工作台
┌──────────────────────────┬─────────────────────────────────────┐
│  左：散件收件箱            │  右：tag 树 (canonical)               │
│                          │                                     │
│  ▸ 随手输入 (采集袋)       │   Tot Talk                          │
│     apple banana run      │     └ Book 1                        │
│                          │        └ Unit 1                     │
│  ▸ 练习派生 (§4.3)         │        └ Unit 2  ◀── 把 dragon 拖这   │
│     dragon🆕 castle🆕     │             cat, dog                │
│     AI:"像'童话'单元?"     │   [+ 输入 tag 路径: Tot Talk›Book1›…] │
└──────────────────────────┴─────────────────────────────────────┘
  · 拖词进节点      = INSERT item_group_member（零拷贝）
  · 输 tag 路径     = 确定性 创建/合并（§4.4）
  · AI 提议成组     = 人确认；从不静默
  · 丢弃一个词      = 它继续留在散件 / 归档
  用户全程只看见认识的 tag —— 没有任何 "level"。
```

工作台干的唯一一件事就是**提升**：决定哪些散件 `LanguageItem` 成为哪个 tag 节点的
正式成员。它不是创建工具——词早就存在，结构化只是加成员关系。

### 4.3 练习派生的候选（承接"练习后的产物"）

收件箱的第二股来源是对话的残留。按 `CLAUDE.md` Rule #3（词数据从 `turn` 文本派生，
不另建事件表），这些是**按需现算、不存储**的：

- 打开工作台时，拿该 learner 的 `turn` 文本 diff 掉当前 tag 树里已有的成员 →
  浮现出"孩子在练习里真的说过、但还没收进任何 canonical 节点"的词。
- 把其中一个收进节点是一次刻意写入：惰性创建 `LanguageItem`（全局去重）+ 插入
  `item_group_member`。

这闭合了产品的核心循环：孩子聊天 → 新词从真实使用中冒出 → 贡献者把好的整理进在长的
书里 → 它们又喂给下一次会话的 Scope。**书从孩子真正说过的话里自下而上生长** ——
这正是产品理念"让新东西悄悄长进来 / 边界一寸一寸往外推"和 Krashen i+1 的具体、
可操作的形态。

### 4.4 `tag_path`：确定性建树

旧的 `levels` 数组逻辑（走一串名字、嵌套、同名兄弟合并）**保留**——它就是"tag 动态
拼成树"。它之所以混乱，问题在*在哪运行、由谁运行*，不在逻辑本身。修法是换触发点、
去掉类型化：

- **输入**：一串有序的精确 tag 名，例如 `["Tot Talk", "Book 1", "Unit 1", "Lesson 1"]`。
- **组装**：走这条路径；每个 tag 在当前 parent 下精确同名匹配已有子节点 → 复用；
  否则新建。词挂在叶节点上。
- **不按位置猜 `kind`。** 节点是无类型的 tag。
- **一份实现**，create 与 update 共用（`groups.py` 里两份分叉的拷贝合并成一个 helper）。

| | 旧（混乱）—— 删 | 新（`tag_path`）—— 留 |
|---|---|---|
| 谁给 tag | AI 从照片瞎猜 | 人确认精确 tag |
| 什么时候 | 采集时 | 整理时 |
| 合并依据 | 模糊（小写/去空格猜） | 精确同名匹配（确定性） |
| 副作用 | 还猜类型化 `kind`（level N） | 无 —— 纯嵌套命名 tag |

## 5. "渐进式"的两个含义——必须分开

- **创作渐进（贡献者）**：一本书跨多次会话建成；词每天落进采集流，定期整理进书。
  节奏由贡献者掌握，取决于源材料拍得多快。
- **消费渐进（跟随者——大众体验）**：书**已经完整**；孩子按计划或掌握度**解锁**
  下一个单元/课次。后续由掌握度追踪管理（V2/V3，`core/mastery.py` stub、
  `LearnerItemStats` 规划）。内容是现成的，前进的只是*暴露范围*。

把这两者混为一谈，是当前混乱的一部分。大众的"渐进式"是在现成内容之上的**消费
渐进**，不是创作。

---

## 6. AI 到底该"接管"在哪

运营者的意图——"家长判断不了英语，让 AI 尽可能接管"——干净地落在两处，
**而不是**每张照片的静默归档：

1. **库已被预先填满。** 跟随者打开一本现成的书，本身*就是*"AI/运营者已经替他
   搞定了"的形态。这是最主要的"接管"。
2. **AI 辅助贡献者整理。** 在工作台里（§4.2），AI 批量抽词、*提议*结构（"这 8 个
   新词像一个童话单元"），在一次专注、可审阅的过程里完成——由人确认。这是刻意且
   可审计的，与"散落在每张照片上、永远差一点、永不收敛"的瞎猜正好相反。

已否决：在每次 ingest 里为每个家长静默推断 `parent_id` / tag 路径 / 树位置。
它服务了错误的人群，且永远做不到可靠。

## 7. 冷启动的现实

"大多数人只是跟一本现成的书"要求库被**先填满**。谁来填？运营者，用第 1 层 +
第 2 层 + AI 辅助整理，产出 1–2 本干净的 canonical 教材（如 `CLAUDE.md` 里点名的
Tot Talk 系列）。在这颗种子存在之前，跟随者路径无书可跟。

这使运营者成为**第一个、也是最重要的贡献者**。因此贡献者工具不是"高级/以后再说"
——它是运营者**现在**就需要、用来给库下种的东西。但它需要的是一个**刻意的建书
工作台**（§4），而不是每次采集时的自动魔法。

## 8. 当前代码要砍/要改什么

| 现在（混乱源） | 改成 |
|---|---|
| `ingest.py` 推断 `parent_id` / `levels` / `book_name` / `unit` / `lesson`，并把整个库塞进 prompt（`groups.py` 既有素材块） | AI **只回 items** + 一个建议 `name` + `cefr`。**不**返回树位置。不再当图书管理员。 |
| `create_group` / `update_group` 的 `levels` 自动建树在 ingest 时运行，两份分叉拷贝 + 位置猜 `kind` | **重用，不是删除。** 把嵌套+合并逻辑留作整理时的 `tag_path` 组装器（§4.4）：精确同名合并、**去掉**位置猜 `kind`、合并成一个 helper、且只由工作台里人确认来触发。 |
| `kind` 作为结构化枚举（`textbook_book/unit/lesson`） | 降级为**可选装饰标签**（§4.1），由 AI `kind_label` 喂。绝不驱动结构；绝不被 Scope 读取。 |
| 一个界面又采集又建树 | 拆成：① **采集**（拍照→确认词→完，约 5 秒）② **整理工作台**（§4；commit `c4bbca7` 已经起头的层级工作台）。 |

注：`get_descendant_group_ids` **保留** —— Scope 仍要遍历 tag 树。它当前的全表扫描
实现是另一个独立的性能清理（改成递归 CTE），以后再做，不属于本次改动。

## 9. 关键路径（按此顺序）

1. **用手工书验证核心循环。** 直接粘贴/塞进 1 本小的 canonical 书——哪怕只是手输的
   1–2 节练习也够（不需要采集管道）。一个孩子跟着学，说 20 轮。盯 Scope 有没有漏
   超纲词；盯它好不好玩。
2. **砍掉/重用自动归档**（§8）。采集变成扁平袋；ingest 只抽词；tag_path 组装器搬进
   工作台。
3. **建整理工作台**（§4）——贡献者把散件（采集 + 练习派生）变成干净 tag 树的工具。
   AI 提议，人确认。
4. **给库下种** 1–2 本 canonical 书（冷启动，§7）。
5. **跟随者路径**：订阅公共书 + 渐进解锁（与 `learner-content-scope.md` 的订阅
   机制、V2 掌握度追踪重叠）。

第 1 步即使素材很少也已解套。第 2 步是清理。第 3–5 步在干净地基上搭"人人跟一本
现成书"的愿景。

## 10. 与其他文档的关系

- `content-model.md`：存储 schema 不变。本文只改变*schema 怎么被使用*（采集袋 vs
  整理后的 tag 树；用 `item_group_member` 做提升；`kind` 降级为装饰）。
- `learner-content-scope.md`：跟随者的只读/订阅行为（§9.4、§2.3）正是这里描述的
  大众消费路径。公共库（§2.7、§12）即第 3 层。
- `architecture.md` §8（教材录入管道）：**修订** —— 管道里"AI 推断层级和
  parent_id"那一步移除；抽取只回 items，结构化是 §4 里那个刻意的工作台动作。
- `CLAUDE.md` 规则 #7（canonical schema）：强化 —— 外部输入仍收敛到
  `Curriculum → Unit → ...` 的形状，但收敛发生在*整理*时、通过 `tag_path`，不是
  *采集*时。
