## 阶段一：身份验证机制的体验打磨

### 1.1 过期会话 (Session Expired) 导致 500 服务器错误的问题修复

在最初的身份校验中，用户登录过期后如果强行访问 `/zh-CN/chat`，系统会因未捕获 401 报错而在服务端抛出 500 崩溃页面。

- **根本原因**：Next.js App Router 的 Server Component 中，`cookies()` 辅助函数是只读的。如果在 Server Component 中直接进行 `cookies().delete()` 清理，会导致写入静默失败。此时 401 重定向到登录页后，由于 Cookie 未清除，中间件 `proxy.ts` 又会再次将其重定向回 Chat 页面，形成死循环，最终导致浏览器报错。
- **最终方案**：
  - 新建 `lib/api.ts`，封装 `createApi()` 工厂函数。一旦遇到 401，主动重定向到 `/{locale}/login?expired=1`（避开在 Server Component 中修改 Cookie）。
  - 中间件 `proxy.ts` 检测到请求参数中带有 `?expired=1` 时，由 Middleware 在响应中强行调用 `response.cookies.delete(COOKIE_NAME)` 进行彻底的 Cookie 清除，随后放行。这优雅地规避了 Server Component 的只读限制。

## 階段二：火山引擎 STT 服务 403 授权故障调试

在进行首次全链路语音转文字（STT）测试时，接口返回了 403 Forbidden 错误。

- **调试轨迹**：
  1. 初始以为是 Resource ID 配置错误。文档中规定 Resource ID 为 `volc.seedasr.sauc.duration`（代表按时长计费的通用英语识别资源）。
  2. 控制台开通服务后依然报 403。经多方比对，确认这是火山引擎旧版控制台反直觉的设计缺陷——**开通服务资源并不等于 APP 自动获得了使用权**。
  3. **解决方案**：在控制台的“APP 密钥管理”中，将开通的语音识别资源实例显式地与当前 APP 的 AppID 进行权限绑定绑定。
- **细节踩坑**：火山语音识别 STT WebSocket 头名称为 `X-Api-App-Key`，而语音合成 TTS HTTP 的请求头为 `X-Api-App-Id`，两者名称各异，必须严防混淆。

## 阶段四：`vocab_event` 数据层架构大重构 (删除旧表)

### 1. 方案博弈与数据爆炸评估

在原构想中，每一轮对话都要记录 `vocab_event` 表（一个单词记录一行），目的是为 V2 掌握度做数据储备。但估算数据量后我们大惊失色：  
*10 万日活跃学习者 × 每天 50 轮对话 × 每轮平均 20 个单词 = 每天产生 1 亿行数据记录！*  
如此庞大的写入量不仅会让小微数据库迅速爆仓，而且在跨轮聚合掌握度时每次都要进行耗时的全表扫描。

### 2. 替代方案的最终科学决策

我们决定**彻底删除 `vocab_event` 表与对应的提取提取类代码**。

- **架构理由**：单词词频属于 `turn.text_user` 与 `turn.text_ai` 的**衍生派生数据**，而非源头原始数据。只要核心对话文本 Turn 仍在，任何时候都可以离线从字符串重算词汇暴露数据。
- **V2 演进方向**：当未来真正上线 Mastery 系统时，引入 **`learner_word_stats` (一学习者、一单词、一行数据)**，使用 `INSERT ... ON CONFLICT DO UPDATE` (Upsert) 进行单行增量数值更新。这样不仅避免了表的数据行数爆炸，查询耗时也从 $O(N)$ 扫描暴降为 $O(1)$。

## 阶段五：V2 全流式架构设计定调

对于流式升级，我们明确了以下重构边界：
1. **录音请求侧保持不变**：依然使用 HTTP 整体上传录音文件。孩子在开口时，会有频繁的停顿、口误或纠错。如果利用静音算法（VAD）自动切断录音，误切率极高。**孩子点击“按停”按钮是最佳的句子切分边界。**
2. **响应侧全面流式**：
   - 收到音频后完成 STT。
   - LLM 开启流式吐字 (`stream=True`)。
   - 累积到整句（遇到逗号、句号、感叹号、问号）立刻切分。
   - 将该短句异步发起 TTS 火山合成。
   - 通过 SSE 管道将生成的音频片段流式推回浏览器。
   - 浏览器接收到便实时拼装播放，达成完美的“流式陪伴”体验。
